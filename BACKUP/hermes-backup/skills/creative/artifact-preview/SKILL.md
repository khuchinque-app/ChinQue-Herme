# Artifact Preview — Visual Delivery for Port-Limited Environments

Use this skill when you create a **visual HTML artifact** (logo, brand card, landing page, UI prototype, deck slide, component lab) and need to show it to the user **without expecting them to open a URL on a random port**.

## Core Rule

**Do not serve HTML files on a random port and expect the user to open it without verification.** The user operates in a text-messaging interface (Telegram). If you serve on a port without checking it first, you will give them a dead URL.

**Two valid delivery modes, pick one:**

### Mode A — Screenshot-only (preferred for design previews)
Render headlessly, screenshot, deliver via MEDIA. No port exposure needed. See workflow below.

### Mode B — Served on port (when user asks for it or is on same network/VPS)
If the user is on the same VPS and asks for a URL, OR if the artifact needs interactivity (JS, animations) that a screenshot can't show:
1. **Check UFW/firewall FIRST** — `sudo ufw status verbose`. If the port isn't open, add it: `sudo ufw allow <PORT>/tcp`
2. **Verify external access** — `curl -s -o /dev/null -w "%{http_code}" http://<PUBLIC_IP>:<PORT>/<PATH>` — must return 200
3. **Take a real browser screenshot as proof** — headless Chromium screenshot of the live URL
4. **Deliver the full URL** — `http://<public-ip>:<port>/<path>` — never a partial URL, never assume the user will add the IP themselves
5. **Send the screenshot alongside** as MEDIA so they have visual proof it works before clicking

## Workflow

1. **Create the HTML file** as normal (self-contained, no remote deps).
2. **Render headlessly** with `chromium-browser`:

```bash
chromium-browser --headless --no-sandbox --disable-gpu \
  --screenshot=/path/to/output.png \
  --window-size=<width>,<height> \
  file:///path/to/your-artifact.html
```

   - Pick viewport dimensions that fit the artifact (e.g. `1000,600` for brand cards, `1920,1080` for full-page designs).
   - Chromium snap sandbox warnings (dbus/AppArmor/mount errors) are non-fatal noise — ignore them if exit code is 0 and the .png file was written.
   - If chromium-browser isn't found, try `chromium` or `google-chrome`.

3. **Verify the screenshot** was written correctly:

```bash
file /path/to/output.png
# Should report: PNG image data, ...
```

4. **Deliver via MEDIA protocol** — include `MEDIA:/absolute/path/to/screenshot.png` in your response. This renders as a native photo (PNG/JPG), video (MP4), or voice message (OGG) in chat.

5. **Mention the HTML path** so the user can request edits to the source file.

## When to Skip Screenshots

- **Non-visual artifacts** (config files, scripts, data exports, code patches) — just report the path.
- **User asks for the file only** — respect that, deliver path only.
- **No browser available** (headless server, minimal container) — fall back to reporting the path and describing the artifact verbally.
- **The artifact is already a native image** (generated by ComfyUI, FAL, etc.) — use MEDIA directly on the original file, no HTML step needed.

## Pitfalls

- **Killing the server you started**: If you started a background HTTP server to serve the file, kill it after taking the screenshot (`kill <pid>`). Don't leave orphan processes.
- **Wrong screenshot dimensions**: If the window is too small, elements clip. Too large and you capture empty space. Match the viewport to the artifact's primary layout.
- **MEDIA path must be absolute**: `/home/user/file.png` works; `~/file.png` and `./file.png` do not.
- **Chromium snap slowness**: First run can take 10-15 seconds. Subsequent runs are faster. Set a generous timeout (30s).
- **Font rendering**: Chromium renders web fonts from Google Fonts etc. when online. If the artifact uses local fonts they may not render.
